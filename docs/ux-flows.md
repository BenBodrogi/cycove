# CyCove — Recovery & Contact-Adding UX

> Design doc for the two flows the no-phone/no-email identity model (see `Projects/CyCove.md` -> Key decisions -> Identity model) makes non-obvious: how you recover an account, and how you add a contact without a phone-number-based address book. Wireframes for these and the core screens are at `docs/wireframes.md` (rendered artifact linked from there). Draft — needs a real design pass before implementation, not implementation-ready.

## Recovery key UX

**The core tension:** no phone/email fallback means a lost recovery key is a lost account — full stop. The UX has one job: make that consequence impossible to miss, without scaring away someone who's just trying to sign up quickly.

### Registration flow
1. Account created (random public ID generated silently, no user input needed).
2. Recovery key generated **client-side** — server never sees it, only a hash (see `docs/api/openapi.yaml` `/register`).
3. Full-screen, non-dismissible-by-accident reveal: the recovery key (a word-list phrase, Bitcoin-seed-phrase-style — easier to transcribe correctly than a random hex string) with:
   - A clear, blunt headline: *"This is the only way to recover your account. We cannot reset it for you."*
   - "Copy" and "I've saved this somewhere safe" as separate actions — copying alone shouldn't auto-advance the flow, to avoid a reflexive tap-through.
   - A forced re-entry step: user types back 3 random words from the phrase before continuing, to catch "I didn't actually write it down" before it's too late — same pattern hardware wallets use for seed phrases, worth reusing rather than inventing something novel here.
4. Confirmation screen: "Recovery key saved. If you lose access to all your devices and this key, your account and messages cannot be recovered." — said once more, plainly, then never blocks the user again.

### Recovery flow (existing account, new/wiped device)
1. "Restore account" entry point on the registration screen (alternative to "Create new account").
2. User enters their recovery phrase.
3. Client re-derives the key locally, requests encrypted key backup from the server using a proof of possession (never the phrase itself) — matches the "recovery key never transmitted" invariant in `THREAT_MODEL.md`.
4. Backup decrypts locally; message history and device identity restore. If cross-signing requires an existing verified device to vouch (see `Projects/CyCove.md` -> Architecture -> Crypto), and none exist (all devices lost), the account restores with a *fresh* trust chain — contacts will see a "identity changed" warning on next contact, same as any E2EE messenger's lost-all-devices case. Worth stating this limitation in-product, not hiding it.

### Open design questions (not resolved by this doc)
- Word-list phrase length/entropy (standard BIP-39-style 12/24 words, or a custom scheme) — a crypto-library decision as much as a UX one, revisit alongside `Architecture -> Crypto`.
- Whether to offer an optional (opt-in, clearly-labeled-as-weaker) "encrypted cloud backup of the recovery key itself, protected by a password" for users who prioritize convenience over the strongest guarantee — this would be a deliberate trade-off, not a default.

**Built (2026-07-20), web only — a real "log back in" and "linked device" flow, though not the full recovery design above (no word-list phrase, no forced re-entry step, no cross-signing):**

The trigger was Ben hitting the literal problem this section describes: close the tab, lose the account, no way back in. Splitting "restore access" into what it actually is turned out to matter for the design — see `.claude/plan.md`'s changelog for the full reasoning, summarized here:

- **Same-device login** (`POST /login`, `CyCoveCrypto.login()`): a username + recovery key reissues a session token for an *existing* `user_id`/`device_id`. This only works because the private keys never left this browser's IndexedDB in the first place — the recovery key here is proving *account* ownership to the server, not reconstructing cryptographic identity, which is a materially different (and much more limited) guarantee than the word-list "recovery flow" sketched above. The client checks for a fresh/empty local key store right after re-attaching (a `KeysUploadRequest` appearing where none should) and refuses to proceed if this browser doesn't actually have the account's keys, rather than silently creating a mismatched identity — pointing the user at device linking instead.
- **Device linking** (`POST /devices/pairing-token` + `POST /devices/link`, both already built server-side in Phase 1 but never wired into the client until now): an already-logged-in device generates a short-lived pairing code (a copy-pasteable blob, same pattern as share codes); a second device pastes it and gets its *own* new keys registered under the same account. This is the actual answer to "access from any device" that the word-list recovery flow above doesn't cover well either — a genuinely new device always needs new keys somehow, whether via linking (this) or full backup restore (still unbuilt). Optionally also supplying the recovery key during linking additionally restores the encrypted contacts backup (unrelated mechanism, same recovery key, reused for convenience) — without it, the linked device works for messaging going forward but starts with an empty contact list.
- **Deliberately not attempted**: reconstructing an account from *only* a recovery key on a device that never held its keys. That's the real "Recovery flow" case sketched above, and needs actual encrypted private-key backup (Matrix's cross-signing + key backup, still an open Phase 1 item — see `Projects/CyCove.md` → Still to build) to do honestly. Faking it by silently generating fresh keys under the old account would be worse than not having the feature.

**Extended (2026-07-20, later same day) — QR-code linking, and a linked-devices list with revoke, prompted by Ben wanting to access CyCove from a phone.** Two things were conflated in that ask, worth separating: "access from a phone" is likely already possible with zero new code if the phone joins the same Tailscale network (see `Projects/CyCove.md` → Hosting) — the real feature here is a QR-code way to run the device-linking flow above, since a long copy-pasted pairing code is genuinely painful on a phone keyboard.

**Does this deviate from "as secure as possible"?** Asked directly, answered directly: the pairing *token* itself doesn't get weaker — same single-use, 5-minute-TTL secret, same requirement that an authenticated primary device generate it. What changes is the real-world threat model: hijacking a copy-pasted code requires reading and retyping a long string, naturally hard to do by accident or from a distance; a QR code can be captured by a camera from across a room in an instant, within that same 5-minute window. The token isn't weaker; the odds of an unintended party capturing it convincingly go up. That's why this pass also built a safety net that didn't exist at all before: a "Your devices" list (`GET /devices`) showing every device on the account with a "Revoke" action (`DELETE /devices/:deviceId`) that ends its sessions immediately and removes it. Without that net, QR alone would have been a real step down from "as secure as possible," not just a cosmetic one.

- **QR display**: `LinkDevicePanel.tsx` renders the same pairing code already shown as text also as a QR image (`qrcode` npm package, client-side only, no network calls) — both stay available, matching how real apps (WhatsApp Web, etc.) offer a QR plus a manual-entry fallback rather than replacing one with the other.
- **QR scanning**: new `QrScanner.tsx` — `getUserMedia` camera access, a `<video>`+`<canvas>` polling loop, `jsqr` (pure JS) decoding each frame. Chosen over the newer native `BarcodeDetector` API specifically because Safari/iOS doesn't support it, and "a phone via a browser" has to mean that too. A successful scan feeds into the exact same `decodePairingCode`/`linkDevice()` path the manual-paste flow already used — no protocol change, just a different way to fill in the same string.
- **Devices list + revoke**: new `DevicesPanel.tsx`, opened via a new "Your devices" button. Revoking your own current device is allowed, not blocked — a legitimate "log out this browser remotely" action — the UI warns clearly rather than preventing it outright.
- **Not testable in this environment**: the browser tooling used for live-testing this session has no real camera, so the actual scan-with-a-phone-in-hand path needs verifying against a real device — QR generation and decoding were verified independently instead (a standalone script using the exact same `qrcode`/`jsqr` libraries confirmed a generated code decodes back to the identical original string), and the manual-paste path the scan feeds into was already proven working live.

**Extended again (2026-07-20, later still) — cross-signing, closing the gap the two sections above both explicitly flagged and deferred ("linked devices inherit no trust").** Ben asked what to build next after confirming QR linking worked well; the honest first answer ("real encrypted key backup") turned out to be the wrong framing on closer inspection — `matrix-sdk-crypto-wasm`'s key-backup feature is for Matrix *room* keys, which CyCove has none of. The actually-applicable, actually-supported feature is cross-signing, which solves two real, already-disclosed gaps at once: a linked device starting unverified to every contact, and safety-number verification only ever covering the one device SAS'd rather than a contact's whole identity.

- **Bootstrap**: every account now generates a master/self-signing/user-signing key triplet (`CyCoveCrypto.bootstrapCrossSigningIfNeeded()`), signs its own device with it, and backs up the private key seeds encrypted under the existing recovery-key-derived key (reusing the exact `/account-data/{dataType}` mechanism the contacts backup already used, just a new `dataType`) — see `docs/crypto-integration-notes.md`'s cross-signing grounding section for why this needs a 4th step beyond the library's own 3-step bootstrap.
- **Linking now actually inherits trust**: a newly linked device that's given the recovery key restores those private keys (`restoreCrossSigningKeys()`); either it or an already-established sibling device then signs the other (`reconcileOwnDevices()`) — the newcomer doesn't sign itself, an already-trusted device vouches for it, same as a real-world introduction. Confirmed live: verified a contact once, linked a second device on the *other* side afterward, and the first side saw it as trusted immediately with no repeat SAS.
- **Verifying a contact now covers their whole identity, not just one device**: `confirmSas()` additionally cross-signs the peer's identity (`OtherUserIdentity.verify()`) once SAS confirms — confirmed via a grounding script that the library does *not* do this automatically on its own, so this is a real addition, not redundant.
- **A real gap found live, not just in theory**: nothing pushes a "a device just linked" signal, so the device generating the pairing code has no way to know *when* to go sign the new one unless it happens to be watching. Fixed pragmatically rather than deferred: the "Link a device" panel polls for a new device while open and signs it the moment it appears; opening "Your devices" also triggers a catch-up reconcile. A device linked while genuinely nothing else is watching stays unsigned until the next time either of those happens — disclosed in `THREAT_MODEL.md`, not hidden.

**Extended once more (2026-07-20, later still) — multi-device message delivery + history sync, closing a gap cross-signing itself exposed rather than solved.** Live-testing cross-signing, a message only reached one of a verified contact's two devices — cross-signing answers "should I trust this device," not "does my client actually send to it." Ben's ask was explicit: every device should be treated the same, and a newly linked device should get the account's existing chat history, not just messages sent from then on. These are two separate mechanisms — fan-out can't retroactively deliver history to a device that didn't exist yet when it was sent, no matter how long anything is retained.

- **Fan-out**: `Contact.deviceIds` replaces the single pinned `deviceId` (`web/src/lib/store.ts`) as the seed/anchor device for verification, but sending no longer reads it — `sendMessage`/`sendTyping`/`sendReadReceipt` (`crypto.ts`) now fetch a contact's *current* device list fresh at send time (`getContactDevices`, reusing cross-signing's own device-discovery plumbing) and encrypt one copy per device. No backend change needed — it's the same `relayToDevice` endpoint, called once per device from the client.
- **History sync is device-to-device transfer, deliberately, not server storage**: CyCove's server deletes every message the instant it's delivered — a stated privacy pillar (`THREAT_MODEL.md`, `PRIVACY_POLICY.md`). Rather than reverse that, an already-established device pushes its own local history directly to a newly linked sibling, encrypted the same way messages already are (`m.cycove.history_sync`, sent to your own userId + the sibling's deviceId — reuses 100% of existing crypto/relay infrastructure). Piggybacks on the exact same discovery mechanism cross-signing already built: when a device notices a new sibling to sign, it now also pushes history to it. Confirmed live: built up real conversation history, linked a second device, and it arrived instantly alongside the inherited trust.
- **A local-storage change history sync depends on**: conversations moved from `sessionStorage` (lost on tab close) to IndexedDB, keyed per-device the same way the crypto library's own store already is — a device can't push history it no longer has.
- **Disclosed limitation, matching the same honesty pattern as everywhere else in this doc**: history only transfers from a device that's actually online around when the new one links. If none ever comes online again, that history isn't recoverable — `LinkDevicePanel.tsx`'s copy says this plainly, and it's recorded in `THREAT_MODEL.md`.
- **A real bug found live, fixed rather than shipped**: the first version saved conversations to IndexedDB fire-and-forget from a React effect; reloading a tab in the narrow window before that write completed silently lost the just-synced history. Fixed by making the save happen synchronously at the point of each mutation (inside the message/history-sync handler, before acking) rather than decoupled into an effect — a reload/crash mid-write now just leaves the message in the server's queue for redelivery next reconnect instead of losing it.
- **Also found and fixed, unrelated to this feature's own code**: a stale `web/.env.local` `NEXT_PUBLIC_API_BASE` override (pointing at a Tailscale HTTPS URL from earlier dev-environment work) made every backend call fail silently after a routine dev-server restart, which looked exactly like a verification regression until traced to its root cause. Repointed at `http://localhost:3000` for local dev; worth remembering next time `NEXT_PUBLIC_API_BASE` needs setting for real Tailscale access.

## Contact-adding flow

**The core tension:** no phone-number-based "find friends from my contacts" — replaced with a QR code / share-link flow, which is less discoverable but more private (see `Projects/CyCove.md` -> What, item 4 restated: no contact-list upload).

### In-person: QR code
1. Each user has a personal QR code (encodes their public ID + a short-lived signing nonce, not raw key material — avoids the code itself becoming a long-lived credential if screenshotted and leaked).
2. Other user scans it via an in-app scanner.
3. Both sides see a "You're now connected with [ID/nickname]" confirmation. This is also the natural moment to layer in safety-number verification (see `THREAT_MODEL.md` — MITM at first contact) as an optional "Verify" step right there, since both devices are already physically present.

### Remote: share-link
1. User generates a link (`cycove.app/add/<token>` — domain placeholder, see `Projects/CyCove.md` -> Hosting) via any external channel (SMS, email, another app) — CyCove doesn't need to be the transport for this step, it only needs to validate the token on the receiving end.
2. Link opens the app (or a lightweight web landing page prompting an app-store install if not installed) and prompts "Add [nickname] as a contact?"
3. Same nonce-based token design as the QR code — single-use or short-TTL, so a leaked link can't be replayed indefinitely.
4. Because this path crosses an external, unencrypted channel (SMS/email), the resulting contact should be flagged as "unverified" until an in-app safety-number check happens — the QR path can reasonably default to a higher trust level since it requires physical proximity.

### Open design questions
- Rate limiting on link/QR generation to prevent abuse (spam-adding) — ties into the same abuse-prevention backlog item in `Architecture -> Backend`; more relevant now that anyone holding your share code can send a request, not just people you've mutually exchanged codes with.

**Built (2026-07-19), web only — resolves the nickname question above and previews the request-based flow, though not the QR/link mechanics (no camera integration, no `cycove.app` domain live yet):**
- Nicknames: **opt-in local-only labels, decided** — never synced to the server (confirmed, no directory added — see `Projects/CyCove.md` → Key decisions → Identity model). A self-chosen username can optionally be embedded in your share code as a *suggestion* for whoever adds you; they can always override it locally. Two separate label concepts, both purely local: what you call yourself (embedded in your own share code) vs. what someone else decides to call you (their own nickname field, wins if set).
- Contact requests: a copy-pasteable share code stands in for the QR code (base64 of `{user_id, device_id, username}`) — no camera, no `cycove.app/add/<token>` link infrastructure yet, both deferred until there's a mobile client to actually scan with. Funnels into the same mandatory-verification gate before messaging — see `THREAT_MODEL.md` → MITM at first contact.

**"Connect directly" removed (2026-07-19), request/accept is now the only add-contact path.** Originally shipped alongside "Send a request" as an immediate, no-accept-step alternative for the in-person-trust case. Live-tested and found to have a real, not-cosmetic, problem: it was purely one-sided — clicking it only added the other person to *your* contact list, with no notification and no reciprocal action on their end. If they hadn't independently also clicked "Connect directly" on you, they had no contact entry to route your side's incoming SAS verification request to at all — the request would go out, but nothing on their screen could ever show it. This surfaced as a confusing, silent verification failure (misreported by the UI as "the emoji didn't match" since the generic cancelled-state copy doesn't distinguish a real mismatch from a request that was never receivable in the first place). The request/accept flow doesn't have this gap — accepting is a real synchronizing network event that updates both sides' contact lists together, guaranteeing whoever verifies next has a place to see it. Share codes stay supported as an alternative to typing a username, just now always going through the request/accept handshake rather than connecting immediately.

## In-conversation actions: reactions, reply, delete, forward, read-receipt opt-out

**Built and live-tested 2026-07-21.** Five per-message actions added to `Conversation.tsx`'s message bubbles, plus a global setting, all riding the same generic to-device relay every other conversation feature already uses — see `docs/crypto-integration-notes.md` for the wire-level design.

- **React:** a fixed row of 6 emoji (👍 ❤️ 😂 😮 😢 🙏), not a full picker — deliberately small and plain, matching this app's minimal styling. Tap the same emoji again to clear your reaction.
- **Reply:** quotes an earlier message above the composer while composing, and above the resulting bubble once sent. If the quoted message isn't available locally (e.g. it was deleted, or history hasn't synced to this device yet), shows a plain "Original message" placeholder rather than erroring.
- **Delete for me:** removes the message from your own view only — purely local, no wire event, nothing sent to anyone.
- **Delete for everyone:** only available on your own sent messages. Tombstones the message on the recipient's devices and your own other devices too (renders "This message was deleted"). A courtesy, not a retroactive guarantee — see `THREAT_MODEL.md`.
- **Forward:** resends an existing message's plaintext to a different (verified, connected) contact — mechanically identical to composing a new message, just pre-filled.
- **Read receipts: on/off toggle** (`Sidebar.tsx` header row): a global switch, not per-contact. Off means `m.cycove.read` is simply never sent — the sender's message status stays at "delivered," never advances to "read." Read state is still tracked locally either way, so flipping the toggle back on mid-conversation doesn't cause a burst of late receipts for messages already "seen."

**Also fixed in the same pass, found while designing delete-for-everyone's own-device consistency:** messages sent from one already-linked device now live-echo to your other already-linked devices, not just to the recipient. Previously, two of your own devices open at the same time genuinely wouldn't show each other's sends — only a *newly linked* device caught up, via the one-time bulk history-sync. See `docs/crypto-integration-notes.md`'s "Chat quick-wins" section for the mechanism.

## More quick wins: message menu, timestamps, contact search, notifications, Escape-to-close

**Built and live-tested 2026-07-21, later the same day.** Ben flagged the Log out button was unreachable (see the fix note below) and asked for 5 more small usability additions, plus said the per-message action row from the entry above was getting crowded.

- **Message actions collapsed into a "⋯" menu**, Messenger-style, replacing the row of 3-5 always-visible text buttons — same actions as before (Reply/React/Forward/Delete) plus a new **Copy** action. Opens as a small dropdown anchored to the button; a full-viewport transparent backdrop closes it on an outside click (same pattern the modal panels already use for their own backdrop-click-to-close, rather than a racy document-level click listener), and Escape closes it too.
- **Message timestamps** — bubbles now show a small time (e.g. "10:27") alongside the existing delivery-status ticks, not just status with no time at all.
- **Contact search** (`Sidebar.tsx`): a text input filtering the contact list by label, purely client-side.
- **Browser notifications**: a new Sidebar toggle (off by default — unlike read receipts, requesting notification permission is something a user should explicitly opt into, not something defaulted on). Only fires when the tab is in the background (`document.hidden`); shows the contact's label, never the message body — see `THREAT_MODEL.md`'s new entry on why that's a narrower bar than FCM's fully-opaque payload, not the same one. Clicking a notification focuses the tab.
- **Escape-to-close on every modal panel** (`AddContactPanel`, `EditContactPanel`, `LinkDevicePanel`, `DevicesPanel`, `ForwardMessagePanel`) via a new shared `useEscapeKey` hook — they already closed on backdrop click, this adds the other standard dismiss path.

**Also fixed, reported live:** the Sidebar's header button row (Add/Link device/Your devices/Read receipts/Log out) overflowed its fixed 280px width with no wrapping, pushing the last button — Log out — off-screen and unclickable. Fixed with `flexWrap`. Verified by checking `document.elementFromPoint()` at the button's actual screen position matched the button itself before *and* after the fix, then actually clicking it. App also now labeled **OPEN ALPHA** in the tab title and on the pre-auth screen, both explicitly requested.
