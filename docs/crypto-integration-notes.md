# Crypto integration notes — matrix-sdk-crypto-wasm

> Ground truth from actually running `@matrix-org/matrix-sdk-crypto-wasm` (v18.3.1) in a throwaway Node script on 2026-07-19 — two `OlmMachine`s, a fake in-memory server, real key upload/query/claim, real Olm-encrypted to-device message, real decrypt, plaintext verified to round-trip correctly. This is why the shapes below are precise rather than guessed from the type definitions alone; `markRequestAsSent` actually validates the response shape against real Matrix API semantics and throws on mismatches, which caught real mistakes during the experiment.

## Why this document exists

`matrix-sdk-crypto-wasm`'s `OlmMachine` is the full Matrix client crypto state machine, not a generic encrypt/decrypt library — its request/response shapes are literally the JSON bodies of Matrix Client-Server API endpoints (`/keys/upload`, `/keys/query`, `/keys/claim`, `/sendToDevice`). CyCove's backend has no concept of Matrix rooms or a real homeserver, so it needs to speak just enough of this JSON shape to satisfy the library, storing/serving the relevant blobs **opaquely** (matching the "dumb pipe" design — the server doesn't need to understand Matrix key formats, just store and forward them).

## The four request types `OlmMachine.outgoingRequests()` produces, and what the backend needs to do with each

### `KeysUploadRequest` → backend's key-upload endpoint
- **Request body** (`req.body`, JSON string): `{device_keys: {...}, one_time_keys: {"signed_curve25519:<keyId>": {...}, ...}, fallback_keys: {...}}`. `device_keys` is one blob per device (self-signed identity keys). `one_time_keys` is a *batch* — the library generates ~50 at once.
- **Required response shape**: `{"one_time_key_counts": {"signed_curve25519": N}}` where `N` is however many unclaimed one-time keys remain server-side for that device after this upload. **`markRequestAsSent` throws if this field is missing** — confirmed by triggering the error directly.
- Backend action: store `device_keys` opaquely per device (replaces the old `identityKey`/`signedPrekey` raw-bytes fields). Store each `one_time_keys` entry as its own row (`keyId` + opaque `keyData` JSON), claimable individually.

### `KeysQueryRequest` → backend's batch key-query endpoint (new — didn't exist before)
- **Request body**: `{"device_keys": {"@user:id": [] | ["DEVICEID", ...]}}` — empty array means "all devices for this user." Issued even for the machine's *own* user right after init.
- **Required response shape**: `{"device_keys": {"@user:id": {"DEVICEID": <device_keys blob>}}, "failures": {}}`.
- Backend action: look up stored `device_keys` blobs for the requested users/devices, return opaquely.

### `KeysClaimRequest` → backend's batch key-claim endpoint (reshapes the existing `/prekeys` claim logic)
- **Request body**: `{"one_time_keys": {"@user:id": {"DEVICEID": "signed_curve25519"}}, "timeout": ...}`.
- **Required response shape**: `{"one_time_keys": {"@user:id": {"DEVICEID": {"signed_curve25519:<keyId>": <keyData>}}}}` — exactly one claimed key per requested device, atomically removed from that device's pool.
- Backend action: this is the *same atomic claim-and-consume semantics already built and verified* for the current `GET /prekeys/:userId` (race-safe `updateMany` claim) — just reshaped to batch multiple user/device pairs per call and to return Matrix-shaped JSON instead of the current flat fields.

### `ToDeviceRequest` → the existing WS relay / `POST /messages`, unchanged in shape
- `Device.encryptToDeviceEvent(eventType, content)` returns a JSON string shaped `{algorithm, ciphertext: {<senderKey>: {body, type}}, "org.matrix.msgid", sender_key}` — this is genuinely opaque ciphertext (Olm output for `type: 0`/prekey messages or `type: 1` for established-session messages).
- **No backend change needed here** — this JSON string (or its base64 encoding) is exactly what goes in the `ciphertext` field of the existing `sendCiphertext`/`deliverCiphertext`/`POST /messages` payloads. The relay already treats this as an opaque blob.

## Receiving and decrypting

`OlmMachine.receiveSyncChanges(to_device_events, changed_devices, one_time_keys_counts, unused_fallback_keys?, decryption_settings?)`:
- `to_device_events` is a JSON **array** (not `{events: [...]}` — confirmed by a "expected a sequence" error when wrapped), each item shaped `{sender, type, content}` where `content` is the `encryptToDeviceEvent` output above.
- `changed_devices` is a `DeviceLists` instance — `new DeviceLists()` (empty) works for a minimal first version; a full implementation would populate `changed`/`left` when a peer adds/removes a device.
- `one_time_keys_counts` is a `Map<string, number>` — an empty `Map()` works for a minimal version; a full implementation reports the *own* device's current OTK counts so the library knows when to generate and upload more.
- Returns `ProcessedToDeviceEvent[]`. A successful decrypt is a `DecryptedToDeviceEvent` with a `.rawEvent` **JSON string** (not a `.content` property directly) shaped `{type, sender, content, sender_device_keys}` — `content` is the original plaintext, byte-for-byte verified round-tripping correctly in the experiment.

## What this means for the Prisma schema

`Device.identityKey`/`Device.signedPrekey` (`Bytes`) → replace with a single `deviceKeys: Json` (the opaque blob). `PrekeyBundle.oneTimePrekey: Bytes` → replace with `keyId: String` + `keyData: Json`, since Matrix one-time keys are identified by string keys like `"signed_curve25519:AAAAHQ"`, not raw bytes.

## Safety-number (SAS) verification — ground truth from a second experiment (2026-07-19)

Same discipline as above: ran the actual interactive verification protocol between two real `OlmMachine`s before building anything. Full sequence, confirmed working end to end (`Device.isVerified()` → `true` on both sides at the end):

1. Requester: `device.requestVerification()` → `[VerificationRequest, ToDeviceRequest]`, `event_type: "m.key.verification.request"`. Relay it.
2. Peer receives via `receiveSyncChanges` → a `PlainTextToDeviceEvent` (verification messages are **not** Olm-encrypted — the security comes from the interactive emoji comparison, not from encrypting the negotiation itself). Peer discovers it via `machine.getVerificationRequests(senderUserId)`.
3. Peer: `verificationRequest.accept()` → `ToDeviceRequest`, `event_type: "m.key.verification.ready"`. Relay.
4. Requester receives it → `verificationRequest.isReady()` becomes `true`.
5. Requester: `await verificationRequest.startSas()` → `[Sas, ToDeviceRequest]`, `event_type: "m.key.verification.start"`. Relay.
6. Peer receives it → `verificationRequest.getVerification()` now returns a `Sas` object.
7. Peer: `sas.accept()` → `ToDeviceRequest`, `event_type: "m.key.verification.accept"`. Relay.
8. **Important:** the next step (`m.key.verification.key`) does *not* come back as a direct return value — it shows up in the regular `outgoingRequests()` queue as a side effect of processing the incoming event. Every incoming to-device event during verification needs a follow-up `outgoingRequests()` poll, not just a single react-to-what-was-returned step.
9. Both sides exchange `m.key.verification.key` this way. After that, `sas.canBePresented()` is `true` on both sides and `sas.emoji()` returns the same 7 `Emoji` objects on both — confirmed identical in the experiment.
10. User compares emoji, then each side calls `await sas.confirm()` → an array of `OutgoingVerificationRequest`s (`m.key.verification.mac`). Relay.
11. `m.key.verification.done` follows via the same "check `outgoingRequests()` after receiving" pattern. Once both sides process it, `sas.isDone()` is `true` and `device.isVerified()` is `true`.

**Architectural consequence:** since verification messages travel as plain `ToDeviceRequest`s — the exact same wire type as encrypted messages (`Device.encryptToDeviceEvent()` also produces a `ToDeviceRequest`, just with `event_type: "m.room.encrypted"` and encrypted content) — the relay should be a **generic to-device event transport** (`event_type` + opaque `content`), not something hardcoded to a single "ciphertext" field. This is a cleaner design than the original messaging-only relay, and it's what actually needs to change in the backend to support verification at all.

## Real bug found building the live UI: per-connection message ordering (2026-07-19)

Building `web/app/ClientApp.tsx` against a real two-tab, two-account run surfaced a genuine backend bug, not a client mistake: the emoji comparison would show identical emoji on both sides (steps 1–9 above all worked), but the confirming side's `sas.isDone()` would never flip to `true` after clicking "they match," even though the other side reached `verified` correctly.

Root cause: `Sas.confirm()` returns *two* outgoing requests back to back — `m.key.verification.mac` immediately followed by `m.key.verification.done` (the ground-truth script above only explicitly names the mac step; the done request riding along in the same `confirm()` call wasn't obvious until logging every `relayOutgoing` call). Both go out as separate WS sends on the same connection, so TCP/WS framing guarantees the *sender's* two frames leave in order. But the backend's `socket.on('message', async (raw) => {...})` handler (`backend/src/routes/connect.ts`) is `async`, and the `ws` library does not await one listener invocation before firing the next — so two `sendToDevice` messages arriving back to back start two concurrent `relayToDevice()` calls, each doing its own async Prisma `create()`. Whichever DB write happens to resolve first pushes to the recipient first, regardless of arrival order. In this run, `done` occasionally reached the peer *before* `mac`, which the peer's `OlmMachine` didn't handle gracefully — it silently never reached `isDone()`.

**Fix:** serialize message handling per WS connection — chain each incoming message onto a `Promise` queue (`queue = queue.then(() => handleMessage(raw))`) instead of relying on the `async` listener callback's return value, which `ws` ignores. Confirmed the fix by running the same two-tab, two-account flow repeatedly (not just once) after the change — both sides reliably reach `verified` every time.

**Why this matters beyond SAS:** any protocol step that depends on two to-device events arriving in the order they were sent — which SAS verification does, and which future features (cross-signing, key backup) likely will too — was silently exposed to this race. It's now closed for good rather than just for the mac/done pair specifically, since the fix serializes *all* message types on a connection, not just `sendToDevice`.

## Generic to-device transport keeps absorbing new features (2026-07-19)

Adding mandatory verification, contact requests, typing indicators, and read receipts on top of the existing message/receipt/verification traffic required zero backend or schema changes — every one of them is just another `event_type` string on the same `m.room.encrypted` to-device wire mechanism, decrypted the same way, distinguished only by the inner `type` field on the decrypted payload (`web/src/lib/crypto.ts`'s `handleIncoming` switch). New types added this round: `m.cycove.typing` (`{state: 'start'|'stop'}`, never persisted — pure ephemeral UI signal), `m.cycove.read` (`{message_id}`, same shape as the existing delivery receipt, different inner type). The repeated get-device-then-encrypt-then-relay pattern across these was extracted into one private `encryptAndSend` helper rather than copy-pasted.

**`m.cycove.contact_request`/`contact_accept`/`contact_decline` are the exception — deliberately *not* on this mechanism, corrected mid-build the same day.** The original design (see the changelog entry two sessions back) put them through `encryptAndSend` like everything else. That's wrong once rate-limiting contact requests specifically becomes a requirement: by the time an Olm-encrypted to-device event reaches the server, it's opaque ciphertext indistinguishable from a real chat message — the server can't single out "contact request" traffic to rate-limit precisely, because the whole point of E2EE is that it can't see what's inside. `contact_request` moved to a plain authenticated HTTP endpoint (`POST /contacts/request`, `backend/src/routes/contacts.ts`) that a normal per-route rate limit applies to; `accept`/`decline` stay on the to-device relay but as genuinely plain (non-Olm-encrypted) content via a new `sendPlain` method, matching how `m.key.verification.*` is already plaintext by design — same reasoning applies (nothing in these three payloads is more sensitive than a username, which the server can already resolve via `/users/by-username` anyway). None of the three ever reach `receiveSyncChanges`/`handleIncoming` now — `ClientApp.tsx` intercepts them by `event_type` before the crypto layer ever sees them, sidestepping an ungrounded question (whether `OlmMachine.receiveSyncChanges` treats arbitrary unrecognized plaintext event types as a generic pass-through the way it does for the known `m.key.verification.*` prefix, or would reject/ignore them) rather than assuming an answer.

**Real bug this surfaced, caught by live testing, not typecheck:** `handleDeclineRequest` (`web/app/ClientApp.tsx`) called `sendContactDecline` without first calling `ensureSessionWith` — unlike the otherwise-identical `handleAcceptRequest`, which does. `encryptAndSend`'s `getDevice()` call requires the local OlmMachine to already have that device's keys cached from a prior key-query; *receiving and decrypting* a message from someone does not itself populate that cache for outbound use. The failure was silent from the outside (clicking Decline visibly did nothing) — traced by adding temporary logging to the handler and `encryptAndSend`, which surfaced the actual thrown error (`No known device ... call ensureSessionWith first`) once triggered via a direct `.click()` call in the browser console (the coordinate-based click tool wasn't reliably reaching the button that session, for unrelated reasons — dispatching the click via JS bypassed that and got a clean repro). Same lesson as the message-ordering bug above: a live two-account test catches exactly the class of bug that "it typechecks and the happy path works" doesn't.

## Usernames became a real directory, not just a local label (2026-07-19)

Reversed the previous session's decision. Usernames shipped as a purely local, embedded-in-your-own-share-code label specifically to avoid building a server-side directory. Ben clarified afterward that the actual intent was "type a username to find and add someone" — share codes are impractical to hand-copy, so without server-side lookup, usernames didn't solve the problem they were built for. `User.username` is now `@unique` in Postgres (lowercase-only, claimed via `PUT /users/username`, resolved via `GET /users/by-username/:username`) — a genuine, if narrow, discoverability surface where there was deliberately none before. Mitigated, not eliminated: rate-limited (`backend/src/lib/rateLimit.ts`) against enumeration/scraping, and restricted to ASCII `[a-z0-9_]` to cap (not eliminate) homoglyph-spoofing risk now that a username is something a stranger can type in and reach you with. See `THREAT_MODEL.md` and `Projects/CyCove.md` → Key decisions → Identity model for the full disclosure.

## Contact backup isn't matrix-sdk-crypto-wasm's key-backup feature (correction found before writing any code, 2026-07-19)

Planned to build encrypted contact-list backup on top of `matrix-sdk-crypto-wasm`'s existing key-backup support (`BackupMachine`, `KeysBackupRequest`, `MegolmV1BackupKey`, `BackupSecretsBundle` — all present in `node_modules/@matrix-org/matrix-sdk-crypto-wasm/pkg/matrix_sdk_crypto_wasm.d.ts`), on the assumption it was a generic "store arbitrary encrypted data on the server" facility, and that this ask was a good excuse to finally build a Phase 1 backlog item ("Cross-signing + encrypted key backup implementation") that had been sitting unbuilt. Checking the actual type definitions before writing any code showed that assumption was wrong: this feature is specifically for backing up Megolm *room* keys, keyed by `RoomId`, so that old group messages stay decryptable after a device loss — a Matrix-rooms concept CyCove doesn't use at all. There's no generic "encrypt and store this blob" method anywhere in it.

**What got built instead:** plain WebCrypto — `PBKDF2` deriving an AES-GCM key from the existing recovery key (already ~244 bits of entropy from two `crypto.randomUUID()`s, so a fixed PBKDF2 salt is an acceptable simplification — the salt exists to defeat precomputation against *low*-entropy human passwords, not a concern at this entropy), used to encrypt/decrypt a JSON blob client-side (`web/src/lib/crypto.ts`: `deriveBackupKey`, `encryptAndUploadContacts`, `fetchAndDecryptContacts`). The server (`PUT`/`GET /account-data/contacts`, `backend/src/routes/accountData.ts`) stores/serves the ciphertext opaquely, same pattern as every other blob in this schema. This has nothing to do with the still-unbuilt Megolm backup feature, which stays exactly as unbuilt as before — this correction didn't accidentally complete that backlog item, just avoided forcing the wrong tool onto an unrelated problem.

## Real bug found live-testing decline: the WS client never reconnected, and dropped sends silently (2026-07-19)

Live-testing the new plain `contact_decline` send (added in the to-device-transport correction above) surfaced a second, unrelated bug: the recipient's own contact list updated correctly on Decline (that state change is local and synchronous), but the original requester sometimes never found out — their UI stayed on "Request sent — waiting…" forever. No error was thrown anywhere; `sendPlain`'s `this.sender` was confirmed set (temporary logging), so the call reached `RelayConnection.sendToDevice`, which called `this.socket.send(...)` unconditionally.

Root cause, confirmed by reading `web/src/lib/ws.ts`: `RelayConnection` opened exactly one `WebSocket` at construction and never reconnected — no `close` handler, no retry. Its `error` handler only logged. Per the WebSocket spec, calling `.send()` on a socket in `CLOSING`/`CLOSED` state is a **silent no-op**, not a thrown exception (only `CONNECTING` throws) — so a decline sent in the window after a connection blip (a real `WebSocket error` was reliably visible in the console immediately before the failed sends, both times this reproduced) vanished with zero signal anywhere: no client-side error, and nothing to relay server-side since the frame never left the browser. The two-account per-message serialization fix above (a *server*-side ordering bug) is unrelated; this one never reaches the server at all.

**Fix:** `RelayConnection` now reconnects itself — `connect()` is a private method the class calls on `close` (unless `close()` was invoked by the caller), with capped exponential backoff (1s → 15s), preserving its public interface (`ready()`/`sendToDevice()`/`ack()`/`close()`) so `ClientApp.tsx`'s one `attachSender` binding stays valid across reconnects without the outer code needing to know. `sendToDevice`/`ack` now check `readyState === WebSocket.OPEN` before calling `.send()`; if not open, the frame is queued in an in-memory outbox and flushed once the new connection's `queueFlush` confirms it's authenticated. Confirmed fixed with a clean repro (fresh account, fresh request, decline, reload all tabs first to guarantee the new code was actually running) — the decline was delivered and the sender's UI updated correctly, repeatably.

**Scope note:** the outbox is in-memory only, lost on tab close/reload same as everything else in this session-storage-based dev harness (see the "related but separate gap" callout in the rate-limiting/backup plan) — it protects against a *transient* connection blip mid-session, not against being offline when a peer acts. That's what the server-side `to_device_queue` (already durable, already deletes-on-delivery) is for; the two mechanisms are complementary, not redundant.

## Deliberately simplified for the first working version (real gaps, not oversights)

- **No live one-time-key-count tracking** — the client should proactively re-upload keys (e.g., check its own remaining count after each claim/decrypt cycle) rather than relying on server-pushed counts over the WS connection. Revisit if key exhaustion becomes a real problem.
- **No device-list-change tracking** — `DeviceLists` stays empty. A device added via `/devices/link` won't automatically be picked up by a peer's `OlmMachine` until that peer calls `updateTrackedUsers`/a fresh key-query. Fine for a first working version with no live multi-device linking test yet; matters more once that flow is exercised for real. **Now actually relevant — see the cross-signing grounding section below**, since this gap directly undercuts "linked devices inherit trust" if a peer never re-queries to see the new device at all.
- **SAS device verification — done (2026-07-19)**, see above. **Cross-signing, key backup, QR verification** — still not wired up. These (along with SAS) are exactly the features that motivated keeping `matrix-sdk-crypto-wasm` over a lower-level library.

## Cross-signing — ground truth from a real script before writing any product code (2026-07-20)

Same discipline as the SAS section above: before designing the `/keys/signing-keys/upload` and `/keys/signatures/upload` backend endpoints, ran real `OlmMachine` instances (two of Alice's devices, one of Bob's, `fake-indexeddb` for the Node store) through the actual cross-signing bootstrap/link/verify sequence. Confirmed a wrong initial assumption along the way: this isn't "Megolm key backup" (that's for Matrix *room* keys, which CyCove has none of — see the correction above) — cross-signing is a separate, real feature this library fully supports and CyCove's gaps actually need.

**1. `bootstrapCrossSigning(false)` returns three requests with confirmed exact shapes:**
- `uploadKeysRequest` — `undefined` if this device already has device keys uploaded (the normal case after `register()`/`login()`), otherwise a normal `KeysUploadRequest` — reuses the existing `/keys/upload` route unchanged.
- `uploadSigningKeysRequest.body` — `{master_key, self_signing_key, user_signing_key}`, each a full Matrix cross-signing key object (`user_id`, `usage`, `keys`, `signatures`). Maps directly onto the new `User.masterKey`/`selfSigningKey`/`userSigningKey` columns — server stores verbatim, same opaque pattern as `Device.deviceKeys`.
- `uploadSignaturesRequest.body` — `{"<user_id>": {"<device_id>": <the device_keys object, but with ONLY the new self-signing-key signature in its `signatures` map — the device's original self-signature is absent>}}`.

**2. Confirmed the hard way that `/keys/signatures/upload` needs a real merge, not a pass-through — a naive overwrite corrupts the device record.** Applying `uploadSignaturesRequest`'s body as a direct overwrite of the stored `device_keys` blob (the same "server never parses, stores verbatim" pattern used everywhere else in this schema) replaced the device's `signatures` map with *only* the new cross-signing signature, discarding its original self-signature (device's own ed25519 key signing itself, present since the very first `/keys/upload`). Once that happened, the device silently disappeared from every subsequent peer's `getDevice()` lookup — a corrupted/under-signed device fails local validation and is dropped, not surfaced as an error. Confirmed fixed by having the endpoint union `signatures[user_id]` entries (new signature added, existing ones preserved) rather than replacing the object outright. This is the one endpoint in the whole backend that can't stay a dumb pass-through.

**3. `bootstrapCrossSigning()` alone does not mark your own device `isCrossSignedByOwner()` — an explicit follow-up `OwnUserIdentity.verify()` call is required.** After bootstrap + applying `uploadSignaturesRequest` (with the merge fix), `alice1.getDevice(self, ALICE_DEV_1).isCrossSignedByOwner()` was still `false`. Calling `machine.getIdentity(ownUserId)` → `OwnUserIdentity.verify()` produces a second `SignatureUploadRequest` (this one signs the *master key* with the device's key, the other direction of the same trust link) — only after uploading *that* did `isCrossSignedByOwner()` flip to `true`. Bootstrap sequence is therefore four steps, not three: upload keys → upload signing keys → upload bootstrap signatures → `OwnUserIdentity.verify()` → upload those signatures too.

**4. A newly linked device does not inherit trust automatically — confirmed the plan's reconciliation step is necessary, not optional.** Simulated linking a second Alice device (`ALICE_DEV_2`) and importing the private cross-signing keys via `importCrossSigningKeys()` (as if restored from the encrypted backup). Alice's *first* device saw `ALICE_DEV_2` as `isCrossSignedByOwner: false` until it explicitly called `.verify()` on that `Device` object (fetched via `machine.getDevice(ownUserId, newDeviceId)`) and uploaded the resulting signature — after which it flipped to `true`, and stayed `true` from a completely independent third party's (Bob's) point of view too, confirmed via a fresh `/keys/query`-equivalent round-trip.

**5. `isCrossSignedByOwner()` is a public, anyone-can-check fact; `isVerified()`/`isCrossSigningTrusted()` are the viewer's own trust decision — confirmed these are genuinely independent.** Bob (a stranger, no prior interaction) could see `isCrossSignedByOwner: true` for both of Alice's devices immediately once the public key material round-tripped through a `/keys/query`-equivalent — no SAS, no trust decision on Bob's part needed for that fact alone. `isVerified()` correctly stayed `false` for both until Bob made his own trust decision. No special serving logic needed beyond what was already planned (serve `master_keys`/`self_signing_keys` to everyone).

**6. The one answer that changes the SAS integration design: `sas.confirm()` does *not* auto-cross-sign the peer's identity, even with cross-signing bootstrapped on both sides.** Ran a full real SAS verification between Bob and Alice's first device (emoji matched: 🐧🐷🐓🔑🌽✏️📎 on both sides, `sas.isDone()` true on both). `sas.confirm()`'s returned array contained only `ToDeviceRequest`s (the mac/done events) — never a `SignatureUploadRequest`. After SAS completed, Bob's view was: `ALICE_DEV_1.isVerified(): true` (the device he actually SAS'd) but `ALICE_DEV_1.isCrossSigningTrusted(): false`, `ALICE_DEV_2.isVerified(): false` (Alice's *other* device, never SAS'd, gets nothing from this), and `getIdentity(alice).isVerified(): false`. Per-device SAS trust and identity-level cross-signing trust are fully independent — confirming the plan's proposed extra step (after `confirmSas()` succeeds against a cross-user device, fetch `getIdentity(peerUserId)` and call `.verify()`, uploading the resulting signature) is genuinely required, not redundant with anything the library already does automatically.

**7. Surfaced the pre-existing "no device-list-change tracking" gap (noted above, dated 2026-07-19) as now directly load-bearing.** None of the above discovery works without the *viewing* machine re-querying a user's devices — `updateTrackedUsers()` alone does not force a re-query for an already-tracked user; it took an explicit `DeviceLists(changed: [userId])` passed into `receiveSyncChanges` to pick up Alice's second device. CyCove's backend has no mechanism today to signal "this contact's device list changed" to already-connected peers. Pragmatic fix folded into the cross-signing implementation rather than treated as a separate project: `ensureSessionWith(peerUserId)` (already called before every message send) now always passes a `DeviceLists(changed: [peerUserId])` refresh for that peer, not just `updateTrackedUsers`, so a stale view can't persist longer than the next message exchange. Not a full sync-driven push mechanism, but closes the gap for the flows that actually matter today (chatting with an existing contact).

Script and fake in-memory server were throwaway (Node + `fake-indexeddb`, not committed) — this section is the durable record of what they proved.

## Multi-device fan-out + history-sync payload size — ground truth before writing product code (2026-07-20, later still)

Two quick, targeted checks before designing multi-device message delivery and device-to-device history sync (the feature prompted by a real gap found live-testing cross-signing: a message only reached one of a contact's two devices, since `Contact.deviceId` is pinned to a single device and never updated).

**1. Confirmed: `OlmMachine.getMissingSessions()` already covers every known device of a tracked user, not just one — no crypto-layer change needed for fan-out.** Set up one account with two devices (`ALICE_DEV_1`, `ALICE_DEV_2`) and a separate account (Bob) that tracks Alice via the same `updateTrackedUsers` + forced `DeviceLists` refresh `ensureSessionWith` already does. `bob1.getMissingSessions([aliceUserId])`'s resulting `KeysClaimRequest` body:
```json
{ "one_time_keys": { "@alice:test": { "ALICE_DEV_1": "signed_curve25519", "ALICE_DEV_2": "signed_curve25519" } } }
```
Both devices are requested automatically in one call — `ensureSessionWith(peerUserId)` (`web/src/lib/crypto.ts`) already establishes sessions with all of a peer's devices once that peer is tracked with a fresh device list (which it now always forces, per the cross-signing fix). The only change fan-out actually needs is on the *send* side: loop over `machine.getUserDevices(peerUserId)` instead of sending to one pinned `Contact.deviceId`.

**2. Confirmed: no practical size wall for bundling ~200 messages into one history-sync to-device event.** A realistic 200-message `ChatMessage[]` bundle (id, direction, body, timestamp, status — same shape as `web/src/lib/store.ts`) serializes to ~40KB raw JSON. Olm encryption adds only base64/ciphertext overhead (roughly +33%, so ~54KB) — trivial against every real limit in the stack: `ToDeviceQueue.content` is an unconstrained Postgres `Json` column, and `@fastify/websocket` (wrapping `ws`) defaults to a 100MB `maxPayload`, unconfigured in this codebase (`backend/src/index.ts`/`connect.ts`) so the default applies. ~200 messages per contact is a safe, comfortable first-pass cap for the history-sync bundle designed in this pass — no encoding-limit surprises expected.
