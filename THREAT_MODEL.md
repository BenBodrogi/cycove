# CyCove Threat Model

> Expands the draft threat model in `Projects/CyCove.md` → Architecture → Crypto (Virdana Obsidian vault). Living document — update as the design changes, and definitely before any public launch or third-party audit. Last updated 2026-07-18.

## Assets to protect

1. **Message plaintext** — the highest-value asset. Must never exist server-side, and should exist client-side only in encrypted-at-rest storage.
2. **Long-term identity keys** (per device) — compromise lets an attacker impersonate a device permanently until cross-signing revocation.
3. **Session/ratchet state** — compromise exposes messages within that session's window; Double Ratchet limits this to a bounded set of messages, not the whole history.
4. **Recovery key** — the only way to restore an account with no phone/email fallback (see the Identity model decision in `Projects/CyCove.md`). Its loss is equivalent to account loss; its theft is equivalent to full account compromise.
5. **Metadata** — who talks to whom, when, how often. Minimized by design (see Metadata Minimization below) but not eliminable — the server necessarily knows *some* routing information to deliver anything.
6. **Push tokens** — FCM tokens, if leaked, could be used to spam wake-signals at a device (denial-of-service / battery-drain annoyance, not a content leak, since the payload carries no message data).

## Trust boundaries

```
[ Device A (trusted) ] --ciphertext--> [ Server relay (untrusted for content) ] --ciphertext--> [ Device B (trusted) ]
```
The server is explicitly outside the trust boundary for content — it is trusted only for *availability* (does it relay/store-and-forward correctly?) and *metadata handling* (does it minimize what it logs/retains?), never for confidentiality of message content. This single line is the design's whole security argument — every mitigation below either protects a device (trusted) or reduces what an untrusted server can learn/do.

## Threat actors

| Actor | Capability | In scope? |
|---|---|---|
| Passive network eavesdropper | Sees ciphertext in transit | Yes — mitigated by TLS + E2EE (defense in depth) |
| Malicious/compromised server operator | Full access to server infra and DB | Yes — this is the core threat the "dumb pipe" architecture is built to survive |
| Server compelled by legal process | Can be forced to hand over whatever the server has | Yes — same mitigation as above; the point is the server has nothing useful to hand over |
| Device thief / malware on one device | Full access to that device's local state | Partially — Post-Compromise Security limits *future* exposure once a new ratchet step occurs; does not protect messages already decrypted on that device |
| Malicious contact (in-conversation adversary) | A legitimate contact who turns adversarial (screenshots, leaks) | Out of scope — no E2EE system defends against a trusted party leaking what they legitimately received |
| MITM at first contact | Intercepts initial key exchange before any trust is established | Yes — mitigated by out-of-band safety-number (SAS emoji) verification, built 2026-07-19 and made mandatory before messaging the same day (see MITM at first contact below) |
| Nation-state / bulk surveillance | Traffic analysis, legal compulsion, infrastructure-level access | Partially — EU hosting (see Hosting in `Projects/CyCove.md`) addresses US CLOUD Act exposure specifically; general traffic-analysis resistance is not a current design goal (no Tor/mixnet-style routing) |

## Threats and mitigations

### Server compromise (full DB + infra access)
- **Exposure without mitigation:** message content, long-term keys, social graph.
- **Mitigation:** server never holds plaintext message content or private keys by design (see `Projects/CyCove.md` → Architecture → Backend data model — `message_queue` holds ciphertext only and is deleted on delivery ack). A full DB dump yields: account IDs, device public keys, prekey bundles (public), transient undelivered ciphertext, and — as of 2026-07-19 — two new things, disclosed here rather than left implicit:
  - **Encrypted contact backup** (`encrypted_backups` table) — opaque ciphertext to the server, same as message content; the decryption key is derived client-side from the recovery key and never transmitted. A DB dump gets ciphertext blobs it can't read, not a plaintext contact list.
  - **Usernames** (`users.username`) — genuinely new, genuinely plaintext. Reversed from the original design specifically to make contacts discoverable by typing a username (see Key decisions → Identity model in `Projects/CyCove.md`); a DB dump now yields a real, if small, identity layer beyond the anonymous account ID — whatever a user chose to make themselves discoverable by. Rate-limited against enumeration in normal operation, but a full DB compromise bypasses rate limiting entirely, same as it would for any table.
- **Residual risk:** undelivered ciphertext for offline recipients does sit server-side until delivered or TTL-expired (see the Free tier limits decision in `Projects/CyCove.md` — 14–30 day queue TTL). A compromise during that window exposes ciphertext (not plaintext) for undelivered messages — acceptable given it's still encrypted, but worth stating explicitly rather than implying zero server-side exposure.

### Contact request/accept/decline are plaintext at the relay layer (new, 2026-07-19)
- **Exposure:** unlike chat messages, `m.cycove.contact_request`/`contact_accept`/`contact_decline` are never Olm-encrypted — see `docs/crypto-integration-notes.md` for why (rate-limiting the actual abuse vector, unsolicited requests, requires the server to be able to see that's what it is, which it structurally can't do for opaque Olm ciphertext). The server sees who's requesting/accepting/declining a contact with whom, and the requester's self-chosen username, in the clear.
- **Mitigation/why this is acceptable:** the only content ever carried is a username, which the server can already resolve for that account via the `/users/by-username` directory anyway — this doesn't disclose anything the username-directory decision above didn't already disclose. The *routing* metadata (which device requested which) is no different in kind from what the server already necessarily sees to relay anything, encrypted or not.
- **Not the same threat as MITM at first contact above** — an attacker seeing "device A requested device B" doesn't let them forge a fake acceptance or intercept the subsequent key exchange; SAS verification (still mandatory before messaging) is unaffected by this.

### Device compromise
- **Exposure:** local plaintext message cache, local private key material, active session state.
- **Mitigation:** encrypted local storage (SQLCipher/Keystore on Android; passphrase-derived IndexedDB encryption on Web — see `Projects/CyCove.md` → Architecture → Client). Double Ratchet's post-compromise security means a new ratchet step (which happens on ~every message) heals the session going forward, even if a specific past step was exposed.
- **Residual risk:** the Web client's browser-storage security posture is explicitly weaker than a mobile OS keystore — already flagged as an open item in `Projects/CyCove.md` → Architecture → Client and worth prioritizing before Web is treated as equally trusted as Android.

### MITM at first contact
- **Exposure:** an attacker who intercepts the very first key exchange (before any device is cross-signed) could establish themselves as a "verified" contact.
- **Mitigation:** out-of-band safety-number verification — comparing a derived 7-emoji sequence (SAS, `matrix-sdk-crypto-wasm`'s `Sas` object) via a channel other than CyCove itself (in person, a phone call, etc.). Built and verified working live 2026-07-19: two real accounts, full interactive verification flow (request → accept → SAS key exchange → identical emoji confirmed on both screens → mac/done → `Device.isVerified() === true` on both sides). **Made mandatory 2026-07-19** — the composer is hidden and replaced with a "verify to start messaging" call to action until `isVerified()` is true for that contact. (The direct-connect add path this used to also cover was removed the same day — see Identity model — leaving the request/accept flow as the only way to add a contact.) This closes the residual "TOFU until someone bothers to click Verify" gap: a contact can be added, but no message content can be sent or received through the normal UI until verification actually completes. Not fully closed: nothing stops a user from independently confirming a *fake* match (clicking "they match" without actually comparing), which is a user-behavior risk no software gate can fully prevent — same limitation every SAS-based E2EE messenger has. **Linked devices inherit trust automatically, resolved 2026-07-20 (later same day) via cross-signing.** When device linking first shipped, a newly linked device was a genuinely separate `Device` row with its own keys and no way to propagate an existing contact's verified status to it — every contact saw it as unverified until independently re-verified. Cross-signing (`OlmMachine.bootstrapCrossSigning()`, `web/src/lib/crypto.ts`'s `bootstrapCrossSigningIfNeeded`/`reconcileOwnDevices`) closes this: an already-cross-signed device signs any sibling device it discovers lacking a signature (`Device.verify()`), and `confirmSas()` now also cross-signs the *whole* contact identity (`OtherUserIdentity.verify()`), not just the one device SAS'd — so a contact verified once shows every current *and future* device of that account as trusted, without a repeat SAS dance. Confirmed live: linked a second device for an already-verified contact and it showed as trusted with zero new verification steps. Real residual gap found and fixed during this pass, not a hypothetical: signing only happens when an already-established device notices a new sibling exists (no server push for "a device just linked"), so `web/app/ClientApp.tsx` polls while the "Link a device" panel is open and also reconciles on-demand when "Your devices" is opened — a device that links while *no* other device is actively watching stays unsigned until one of those next happens. See `docs/crypto-integration-notes.md`'s cross-signing grounding section for the full mechanism, including a security-relevant finding: `/keys/signatures/upload` can't be a pure opaque pass-through like every other endpoint — a naive overwrite of stored signatures silently drops a device's own self-signature and corrupts the record, so that one endpoint merges instead.

### Unauthorized device linking via QR capture
- **Exposure, added 2026-07-20 when QR-code linking shipped:** the pairing token behind device linking hasn't gotten weaker — same single-use, 5-minute-TTL secret as the copy-paste flow it's an alternative encoding of — but the real-world odds of an unintended party capturing it go up. Reading and retyping a long pairing-code string is naturally hard to do by accident or from a distance; a QR code displayed on a screen can be captured by a phone camera from across a room in an instant, within that same 5-minute window. Directly asked by Ben whether this deviates from "as secure as possible" — the honest answer is that QR alone, without a mitigation, would have been a real step down, not just a theoretical one.
- **Mitigation:** a linked-devices list with revoke (`GET /devices`, `DELETE /devices/:deviceId`, `DevicesPanel.tsx`'s "Your devices"), built in the same pass specifically because QR raised the stakes enough to require it. Revoking a device invalidates its live sessions immediately (`revokeDeviceSessions` in `backend/src/lib/auth.ts`) and deletes its row (cascading to its keys and queued messages). This doesn't prevent a captured QR code from being used once — the 5-minute window is still real — but it means an unauthorized link is discoverable (the device shows up in the list, unverified, same as any new device) and reversible, not a silent, permanent compromise.
- **Residual gap:** nothing currently *notifies* the account owner when a new device links — discovering an unwanted one still requires opening "Your devices" and noticing something unfamiliar, rather than a push/in-app alert at the moment it happens. Worth a future pass; not built this round.

### Multi-device history sync depends on another device being online
- **Design, added 2026-07-20 alongside multi-device message fan-out:** a newly linked device receives the account's existing chat history via a direct, encrypted device-to-device transfer (an already-established device pushes its own local history to the new one, over the same relay messages already use) — not via the server, which continues to delete every message the instant it's delivered, unchanged from the zero-retention design stated elsewhere in this document and in `PRIVACY_POLICY.md`. This was an explicit choice, not an oversight: the alternative (the server durably storing encrypted message history so any device can fetch it later) would be more reliable but is a real reversal of "the server never retains message content" — not taken.
- **Real limitation, not hidden:** history only transfers from a device that's actually online at or after the moment a new device links. If no prior device ever comes online again, that history is not recoverable on the new device — it still receives messages sent from then on (fan-out doesn't depend on history sync), it just starts with no prior context. Disclosed directly in `LinkDevicePanel.tsx`'s copy at the moment it matters, same pattern as every other disclosed limitation in this document.
- **Mitigation for the message-loss variant of this, found live rather than theorized:** early in testing, a message could be lost locally if a tab reloaded in the narrow window before its IndexedDB write completed (a fire-and-forget save racing a reload). Fixed by making the local save complete *before* acknowledging the message to the server (`applyConversationsUpdate` in `web/app/ClientApp.tsx`) — a reload or crash mid-write now leaves the message in the server's queue for redelivery on next reconnect (the existing per-device queue-and-delete-on-ack mechanism, see `backend/src/lib/relay.ts`) instead of silently dropping it.

### "Delete for everyone" is a courtesy, not a guarantee
- **Design, added 2026-07-21 alongside reactions/reply/delete/forward:** deleting your own sent message "for everyone" tombstones it on the recipient's devices and your own other devices (`m.cycove.delete`, same generic to-device relay as everything else — no server involvement beyond routing the event, same as any other message). It is **not** retroactive protection: if the recipient already read the plaintext, screenshotted it, or otherwise captured it before the delete event arrives, deleting it here cannot undo that — same limitation every mainstream messenger with this feature has, disclosed here rather than implied to be stronger than it is.
- **Ordering edge case, theoretical rather than observed:** a delete event could in principle arrive at a device before the original message it targets (e.g. the original was queued while offline and the delete was sent later but processed first on reconnect). The existing per-connection FIFO message-ordering fix (see the SAS mac/done ordering fix earlier in this document's history) makes this unlikely in practice for a single sender's own event stream, but it isn't formally guaranteed by the current implementation — a delete event referencing a message ID that hasn't arrived yet is simply a no-op (`Conversation.tsx`'s message list won't have a matching id to tombstone), not a crash, and the original message would still render normally if it arrives afterward.

### Recovery key loss or theft
- **Loss:** irrecoverable account loss — no phone/email fallback exists by design. This is a genuine UX/support problem, not just a security footnote — see the Identity model section's open recovery-UX question in `Projects/CyCove.md`. The threat model's job here is to make sure the trade-off is *designed in deliberately*, not discovered by a support team after the fact.
- **Theft, revised 2026-07-20 now that `POST /login` exists:** a stolen recovery key + username is **not** full account takeover on its own — `CyCoveCrypto.login()` refuses to proceed on any browser that doesn't already hold the account's real local key material (detected via an unexpected `KeysUploadRequest` right after re-attaching; see `docs/ux-flows.md`), so an attacker with just the recovery key text, on their own device, gets a clear refusal, not a working session. The real exposure is narrower but still real: a stolen recovery key **plus** access to the same browser/device the account was created on (or already logged into) is full compromise — same as it already was before `/login` existed, since anyone with that level of access already had the live session token too. `/devices/link` is a separate, smaller exposure worth naming explicitly: a stolen recovery key lets an attacker link a *new* device to the account from anywhere (no local-key requirement, by design — that's the whole point of linking) and, if they also have the key, restore the contacts backup onto it. That new device would show as unverified to existing contacts, same as any new device — not a silent, undetectable compromise, but real enough that the recovery key deserves the same "never transmitted, never stored" invariant treatment as before, now extended to two capabilities instead of one. Mitigation unchanged: the recovery key should never be transmitted anywhere except this once, and the server only ever receives a hash of it (`recovery_key_hash`, same convention for `/register` and `/login`).

### Metadata exposure
- **What the server unavoidably knows:** account existence, device count per account, coarse timing of message delivery (when a device came online and fetched something), storage-quota usage.
- **What it's designed not to know:** message content, precise social graph beyond "these two device IDs exchanged ciphertext at some point" (no contact-list upload — see the What section of `Projects/CyCove.md`).
- **Mitigation:** sealed-sender-style routing (server routes by recipient device endpoint without needing sender identity attached) — drafted, not yet implemented. No plaintext logging at any layer, ever — this should be a code-review-enforced rule, not just a design statement (see Conventions to follow when extending in `Projects/CyCove.md`).

### Push notification abuse
- **Exposure:** a leaked FCM token could be used to trigger wake-signals at a target device (battery drain, minor DoS-style annoyance).
- **Mitigation:** push payload is opaque (no content, no identifiers) by design, so the worst case is a nuisance wake-up, not a data leak. Rate-limiting push triggers server-side is worth adding to the abuse-prevention backlog (see `Projects/CyCove.md` → Architecture → Backend open questions).

## Out of scope (explicitly, so it's a decision and not an oversight)
- Defending against a legitimate contact who leaks what they received (screenshots, forwarding) — no E2EE system solves this.
- Traffic-analysis resistance against a global passive adversary (no onion routing / mixnet). EU hosting mitigates *legal* compulsion exposure, not *technical* traffic-analysis capability — worth being honest about this distinction in any public-facing security claims.
- Endpoint compromise via OS-level exploits (root/jailbreak-level attacks) — standard "assume the OS sandbox holds" boundary, same as virtually every mobile app.
